# 🎮 ProjectD: 탑뷰 익스트랙션 슈터 협업 가이드

본 가이드는 **ProjectD** 개발 시 바이너리 파일(블루프린트, 맵)의 충돌을 방지하고, 효율적인 협업을 위해 깃 마스터가 작성한 필수 수칙입니다. 모든 팀원은 작업 시작 전 본 내용을 반드시 숙지해주시기 바랍니다.

---

## 🚩 1. 개발 환경 초기 세팅 (최초 1회)
소스코드를 내려받기 전, 대용량 파일 관리를 위한 설정이 선행되어야 합니다.

1. **Git LFS 설치**: [git-lfs.github.com](https://git-lfs.github.com/)에서 설치 파일을 내려받아 실행합니다.
2. **LFS 활성화**: 터미널(CMD/Git Bash)에서 아래 명령어를 입력합니다.
   ```bash
   git lfs install
   ```
3. **저장소 클론**:
   ```bash
   git clone https://github.com/ProjectDNB/ProjectD.git
   ```
   * *에셋 파일이 1KB 텍스트로 보인다면 `git lfs pull`을 실행하세요.*

## 📂 2. 폴더 구조 및 작업 영역 구분
우리 프로젝트의 `Content` 폴더는 용도에 따라 두 가지 영역으로 엄격히 분리 운영합니다.

### A. 개인 개발자 폴더 (`Content/Developers/[본인이름]`)
* **용도**: 신규 기능 프로토타이핑, 개인 테스트 맵, 임시 에셋 보관.
* **권한**: 락(Lock) 없이 자유롭게 작업 가능합니다. 
* **주의**: 다른 팀원의 폴더는 절대 수정하지 않으며, 이 폴더 내의 파일은 언제든 삭제될 수 있는 임시 파일로 간주합니다.

### B. 메인 프로젝트 폴더 (`Content/ProjectD`)
* **용도**: 게임에 실제로 반영되는 확정된 공용 에셋(블루프린트, 캐릭터, 웨폰 등).
* **권한**: **반드시 락(Lock) 프로세스를 준수**해야 수정이 가능합니다.

---

## 🔒 3. 수정 및 락(Lock) 프로세스
바이너리 파일(`.uasset`, `.umap`)은 Merge가 불가능합니다. 따라서 수정 전 권한 획득이 필수입니다.

### [신규 기능을 만들 때]
1. 개인 폴더(`Developers/이름`)에서 로직을 완성합니다.
2. 검증된 파일을 메인 폴더(`ProjectD`)의 적절한 위치로 **이동(Move)** 시킵니다.
3. 해당 폴더 우클릭 후 **[Fixup Redirectors (리다이렉터 수정)]**를 실행한 뒤 커밋/푸시합니다.

### [기존 메인 에셋을 수정할 때]
1. **락(Lock) 획득**: 언리얼 에디터에서 해당 에셋 우클릭 -> **[Check Out]**을 선택합니다.
   * 이미 다른 사람이 체크아웃했다면 아이콘에 **빨간색 X** 또는 다른 팀원의 이름이 표시됩니다. 이 경우 수정하지 말고 기다리세요.
2. **작업 및 저장**: 체크아웃 성공(아이콘에 체크 표시 확인) 후 작업을 진행합니다.
3. **푸시 및 해제**: 작업을 마친 후 `git push`를 성공하면 자동으로 락이 해제됩니다. (수동 해제 시 `git lfs unlock [파일경로]`)

## ✍️ 4. 커밋(Commit) 및 참조 규칙
### 1. 커밋 메시지 형식
`[태그] 내용 (Jira 티켓 번호 - 선택)`
예: `[FEAT] 캐릭터 기본 이동 C++ 로직 구현`

### 2. 주요 태그 설명
- **[ADD]**: 신규 에셋/파일 추가 (최초 1회)
- **[FEAT]**: 새로운 기능 완성 및 로직 구현
- **[FIX]**: 버그 수정 및 에러 해결
- **[UPD]**: 기존 기능 강화, 수치 조정, 에셋 교체
- **[DOC]**: 코드 주석 및 문서 작업
### 그 외 사항
* **Jira 연동**: 커밋 메시지 서두에 Jira 티켓 번호를 반드시 포함하세요.
  * 예: `[PROJ-10] 총기 반동 로직 및 카메라 쉐이크 구현`
* **상호 참조 금지**: 메인 폴더(`ProjectD`)의 파일이 개인 폴더(`Developers`)에 있는 파일을 참조하게 만들지 마세요. 개인 폴더 정리 시 메인 프로젝트의 레퍼런스가 깨집니다.
* **C++ 코드**: 코드는 락이 필요 없으나, 푸시 전 반드시 로컬에서 **컴파일 성공** 여부를 확인하세요.

---

### 🌿 5. 브랜치 운영 규칙 (ProjectD)

1. **개발자 개인 브랜치는 생성하지 않습니다.** (예: `dev/jaehyuck` 금지)
2. **C++ 코드 기반의 기능 구현**은 `feature/[기능명]` 브랜치를 생성하여 작업합니다.
- 작업 완료 후 로컬 컴파일 성공 시 `develop` 브랜치로 머지 요청(PR)을 보냅니다.
3. **블루프린트, 맵, 데이터 작업**은 별도 브랜치를 파지 않고 `develop` 브랜치에서 직접 수행합니다.
   - **이유**: 바이너리 파일은 브랜치 간 머지가 불가능하기 때문입니다.
   - **대신**: 반드시 수정 전 **락(Lock/Check-out)**을 먼저 획득한 후 작업하십시오.
